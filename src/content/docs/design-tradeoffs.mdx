<Lead>
  很多开发者第一次看 Vyper，会先注意到它“少了很多东西”。但从文档视角看，
  这些缺失恰恰定义了 Vyper 的价值主张：减少容易出错的自由度，让合约行为更显式。
</Lead>

<h2 id="why-less-is-more">为什么更少即更多</h2>

Vyper 在设计上持续围绕三个关键词展开：

- 安全性
- 简洁性
- 可审计性

因此它会主动移除那些虽然强大、但会让控制流和状态边界变模糊的能力。
这不是为了限制开发者，而是为了让重要逻辑更接近“读代码就能看懂”的状态。

<h2 id="solidity-comparison">与 Solidity 的核心差异</h2>

| Solidity 习惯 | Vyper 做法 | 背后的理由 |
| --- | --- | --- |
| modifier | 内联 `assert` / `raise` | 检查逻辑留在函数体里，执行顺序更直观。 |
| 继承 | `import` + `initializes` + `exports` | 组合关系更显式，没有线性化复杂度。 |
| inline assembly | 内建低层函数 | 保留底层能力，但减少不可读 opcode 区域。 |
| 无限循环 / 递归 | 有上界循环，不支持递归 | gas 成本更容易分析。 |
| 隐式外部调用 | `extcall` / `staticcall` | 危险边界在语法上可见。 |

`refer/docs/solidity-differences.rst` 还强调了一点：Vyper 对类型转换更严格，
这会让你多写一些 `convert()`，但也显著减少“调用点看不出风险”的情况。

<h2 id="style-guidelines">风格建议</h2>

`refer/docs/style-guide.rst` 虽然主要写给 Vyper 编译器项目本身，但里面很多原则同样适用于合约代码：

- 命名尽量短、明确、可搜索。
- 布尔值优先使用正向命名，例如 `is_active`，避免双重否定。
- 一个函数只负责一件事，让测试和审计都更集中。
- 新能力上线时，同步补测试和文档，不要让“代码是真相，文档过期”长期存在。

对合约仓库来说，最重要的风格不是排版，而是结构是否稳定：文件职责清不清楚、命名能不能快速定位、失败路径是否明确。

<h2 id="documentation-habits">文档习惯</h2>

原始风格指南还给出了很实用的文档写作建议：

- 用现在时和祈使式描述 API 行为。
- 保持术语前后一致。
- 每段只讲一个主题。
- 新功能不应脱离文档单独落地。

<Callout title="落地方式">
  最可靠的做法不是“最后补文档”，而是在新增模块、外部函数或底层调用时，
  同步记录它的用途、权限边界和失败条件。这样文档才会真正服务于审计和维护。
</Callout>
